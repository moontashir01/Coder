# Roadmap: Making Coder a Professional Agent

Recommendations to move Coder from a working demo to a trustworthy, professional-grade
offline coding agent. Scoped to **exclude the GUI (Phase 3)**. Every item below is grounded
in the current codebase, not generic advice.

Ranked by leverage — Tier 1 items give the biggest jump in agent quality.

---

## Tier 1 — Reliability (do these first) — ✅ DONE 2026-07-05

### 1. Verify-and-repair loop after every code action ✅
> Shipped: `app/agent/verify.py` (`check_file`: .py in-process compile, .js `node --check`,
> .ts `tsc --noEmit`, .html tag-balance; missing checker = unverifiable-ok) +
> `AgentCore._verify_and_repair` wired into `_file_op_flow` and `_surgical_edit`,
> capped at `settings.max_repair_attempts` (default 2). Tests: `tests/test_verify.py`.
**Problem.** The agent writes files and never checks that they work. This is the single
biggest gap between a toy and a professional agent. `run_command` already exists but is not
wired into an autonomous check loop for file operations.

**Fix.** After generating/editing a file, run a cheap correctness check and feed failures back
to the model for repair:
- `.py` → `python -m py_compile <file>`
- `.js` → `node --check <file>`
- `.ts` → `tsc --noEmit`
- `.html` → basic structural / tag-balance validation

Loop: generate → check → on error, send the error back → model repairs → re-check
(cap at N iterations). This directly fixes the multi-file website case — broken JS/HTML gets
caught and corrected instead of shipped.

**Impact.** Highest. Turns "it wrote something" into "it wrote something that runs."

---

### 2. Native tool-calling instead of hand-rolled JSON ✅
> Shipped: `_run_tool_loop` uses `ChatOllama.bind_tools(registry.to_openai_tools())` +
> `AIMessage.tool_calls` + `ToolMessage` feedback. Deleted `_tool_loop_prompt`,
> `_parse_action`, `_normalize_action`, `_coerce_args`. Kept Tool-not-found correction +
> §11 recovery/give-up.
> Live smoke 2026-07-05 (qwen2.5-coder:7b): the installed **Ollama 0.31.1 is too old to emit
> structured `tool_calls`** (tool JSON arrives as plain content, even via the raw API). Added a
> narrow `_parse_textual_tool_call` fallback (whole-content `{"name","arguments"}` only); smoke
> then passes end-to-end (write_file executed, file on disk). **Recommended: upgrade Ollama**,
> after which the fallback stops firing.
**Problem.** The `_run_tool_loop` machinery — `_tool_loop_prompt()`, `_normalize_action()`,
`_coerce_args()`, the "Tool not found" correction, retry-on-unparseable — all exists because
the old 3B model mangled the JSON tool protocol.

**Fix.** Ollama + `qwen2.5-coder` now support real function-calling. Use
`ChatOllama.bind_tools()` and let the model emit structured tool calls natively. This lets us
delete most of the brittle JSON-repair code and get far higher reliability. Low risk on 7B.

**Impact.** Biggest simplification + robustness win in one change. Removes a whole class of
"model emitted almost-valid JSON" failures.

---

### 3. Fix cross-file consistency in `_multi_file_flow` ✅
> Shipped both halves: a plan **manifest** injected into every per-file call AND
> sequential generation passing already-written sibling contents via the new
> `extra_context` param on `_file_op_flow`/`_surgical_edit`.
**Problem.** Each planned file is generated by an **independent** LLM call
(`_file_op_flow` per file). The call that writes `index.html` never sees the generated
`styles.css` or `script.js`, so `<link href>` / `<script src>` references and shared class
names are not guaranteed to line up.

**Fix.** Either:
- Generate files **sequentially**, passing already-generated siblings as context to each
  subsequent call, **or**
- Have `_plan_file_ops` emit a shared **manifest** (agreed filenames + a small interface
  contract: class names, function signatures, IDs) and inject it into every per-file
  generation.

**Impact.** Makes multi-file output an actually-working project instead of loosely related
files.

## Tier 2 — Quality

### 4. Fix RAG chunking (tree-sitter)
**Problem.** `tree-sitter 0.25.2` + `tree-sitter-languages 1.10.2` are incompatible in this
env; `_chunk_with_tree_sitter` silently falls back to token-window chunking. Semantic chunking
is **not actually running** (documented gotcha in CLAUDE.md).

**Fix.** Pin `tree-sitter==0.21.3`, or migrate to `tree-sitter-language-pack`. Semantic
(function/class) chunks → better retrieval → better answers and edits.

---

### 5. Token-aware context management
**Problem.** `max_context_tokens` is defined in `config/settings.py` but **read nowhere** in
the code. History trimming is a fixed message count (`conversation_buffer_size = 20`), with no
token budget. Long sessions can silently overflow the model's context window.

**Fix.** Add token-based history truncation against `max_context_tokens`, plus conversation
summarization for older turns. Project summarization already exists — extend the same idea to
conversation history.

---

### 6. Offline eval harness
**Problem.** The 3B → 7B model swap shipped with **zero measurement** of whether quality
improved. There is no way to detect a regression when changing models or prompts.

**Fix.** Build a small offline golden-task suite (10–20 prompts) that asserts observable
outcomes: file created, edit applied, answer contains an expected token, plan has N files, etc.
Run it on every model/prompt change.

**Bonus.** Once evals exist, the 3B-era hardening (see the flagged block in CLAUDE.md) can be
loosened with data instead of guesswork.

---

## Tier 3 — Polish

### 7. Wire up streaming
`get_streaming_llm()` builds `self._llm_stream` (core.py) which is then **never used** — dead
code. langchain_ollama streams via `.stream()` / `.astream()` at the call site. Wire it into
REPL output so tokens stream as they generate. Pure UX win.

### 8. Real permission gating + safe writes
`ToolDefinition.permissions` exists but is unused; `allowed_commands` is informational only
(only `blocked_commands` is enforced). `write_file` overwrites existing files freely. Add:
- diff preview before a mutating write,
- an undo / backup step,
- actual enforcement of the `permissions` field.

### 9. Packaging
Ship a `pyproject.toml` with a console entrypoint (`coder`) so it installs via `pipx install`
instead of `python main.py`. Add versioning.

---

## If you do only three

1. **#1 Verify-and-repair loop** — correctness.
2. **#2 Native tool-calling** — reliability + big code deletion.
3. **#6 Eval harness** — so you can *prove* 1 and 2 helped.

Do **#6 first** — it's the measuring stick for everything else.
